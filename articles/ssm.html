# 为何不添加 context:annotation-config

<context:annotation-config/> 的作用为隐式的向Spring容器注册

​ AutowiredAnnotationBeanPostProcessor,

​ CommonAnnotationBeanPostProcessor,

​ PersistenceAnnotationBeanPostProcessor,

​ RequiredAnnotationBeanPostProcessor

这4个BeanPostProcessor.注册这4个bean处理器主要的作用是为了你的系统能够识别相应的注解。

值得注意的是<context:component-scan/>不但启用了对类包进行扫描以实施注释驱动 Bean 定义的功能，同时还启用了注释驱动自动注入的功能（即还隐式地在内部注册了 AutowiredAnnotationBeanPostProcessor 和 CommonAnnotationBeanPostProcessor），因此当使用 <context:component-scan/> 后，就可以将 <context:annotation-config/>移除了。

当加入了包扫描器后，就无需添加注解配置器

 <!--事务控制-->
    <bean id="transactionManager" class="org.springframework.jdbc.datasource.DataSourceTransactionManager">
        <!-- 控制数据源 -->
        <property name="dataSource" ref="pooledDataSource"/>
    </bean>
    <!-- 基于注解的事务开启，使用xml配置形式的事务 -->
    <aop:config>
        <!-- 切入点表达式 -->
        <!-- public(可以不写) 返回值类型所有类型 包下的所有的方法包括子包，方法的参数任意多-->
        <aop:pointcut expression="execution(* com.fr.service..*(..))" id="txPonit"/>

        <!-- 配置事务增强，关联事务切入点表达式，切入点如何切入 -->
        <!-- 让切入点表达式和事务增强联系在一起 -->
        <aop:advisor advice-ref="txAdvice" pointcut-ref="txPonit"/>
    </aop:config>

    <!-- 配置事务增强，事务如何切入 -->
    <tx:advice id="txAdvice" transaction-manager="transactionManager">
        <tx:attributes>
            <!-- 所有方法都是事务方法 -->
            <tx:method name="*"/>
            <!-- 以get开始的所有方法 -->
            <tx:method name="get*" read-only="true"/>
        </tx:attributes>
    </tx:advice>

# 数据库表的设计原则: 范式

# 候选码(候选键)和主属性的关系

候选码是可以唯一标识表中的某个属性或某个属性组

而主属性是候选码中某一个属性

理解: 如果有一个学生选课表

Stu_id(学号) Course_id(课程号) Score(成绩)
1 1001 60

学号和课程号可以唯一确定成绩，但是都不能单独确定成绩，则学号和课程号的组合可以成为候选码，而学号和课程号都可以称为主属性。对于上表，我们可以根据学号查询出学生的全部信息，根据课程号查询出课程的全部信息

候选码可能有多个，则选择其中的某一个作为主键。

# 平凡函数依赖和非平凡函数依赖

平凡函数依赖： 任何一个X和Y的组合都可以推出X->Y，例如(学号，性别) -> 学号,(学号，性别) -> 性别

非平凡函数依赖: 如果X -> Y 且 X 包含 Y ，例如 (学号) -> 姓名, 这就是非平凡函数依赖

# 部分函数依赖

X -> Y

X 是组合属性, Y依赖于X中的部分

例如,(学号, 课程名称) → (姓名, 年龄, 成绩, 学分)

其中 姓名、年龄只依赖于学号，学分 依赖于课程名称

# 传递函数依赖

# 第一范式

定义: 每个表有唯一的主键，且每个字段是原子的，不可再分的

# 第二范式

定义: 每个表中的非主属性必须完全依赖主属性，不能出现部分依赖。如果存在，那么需要将部分依赖主属性的属性和该主属性单独抽出，建立新的表

如果一个表中只存在一个主键，那么其一定不会出现部分函数依赖

目的: 消除表中非主属性对主属性的部分函数依赖

# 第三范式

定义: 每个表中的非主属性应该只和主属性有相关性，非主属性之间应该是独立无关的。

(学号) -> (学院) -> (学院地点，学院电话)

学院依赖于学号，学院地点和学院电话依赖于学院，学院地点和学院电话传递依赖于学号

理解: 学院可以决定学院地点和学院电话，其中学院是非主属性，而学院地点和学院电话依赖于学院，所以不满足第三范式中非主属性之间应该是独立无关的，所以可以将学院，学院地点，学院电话单独抽出成一张新表，即学院表，而学号只和学院有关。

目的: 消除表中非主属性对主属性的传递函数依赖

# BC范式

任何一个BCNF必然满足：

  • 所有非主属性都完全函数依赖于每个候选键

  • 所有主属性都完全函数依赖于每个不包含它的候选键

  • 没有任何属性完全函数依赖于非候选键的任何一组属性

目的: 消除表中主属性的传递函数依赖

例子:

有一个配件管理表

一个职工只能在一个仓库中，一个职工可以制造多个配件，一个配件只能由一个职工负责

职工号 配件号 仓库号 数量
1 1001 1 16
1 1002 1 12
2 2003 2 1
3 2004 2 2

其中，1号职工只能出现在1号仓库，管理1001和1002两种配件，2号职工和3只能出现在2号仓库，管理2003和2004两种配件。

(职工号) -> (仓库号) 根据职工号即可以知道其所在的仓库号

(配件号,仓库号) -> (职工号) 根据某一个仓库下某一种配件，就可以找到某一个职工

(职工号,配件号) -> (数量) 根据职工号和配件号，可以找到该职工所负责的配件的数量

将上表进行分解

职工所属仓库表

职工号 仓库号
1 1
2 2
3 2

职工配件表

职工号 配件号 数量
1 1001 16
1 1002 12
2 2003 1
3 2004 2

create table admin_(
	id int(11) not null auto_increment comment '管理员id',
	username varchar(32) comment '管理员账号名', 
	password varchar(50) comment '管理员密码',
	primary key(id)
)auto_increment=1;

create table category_(
	id int(11) not null auto_increment comment '分类id',
	type varchar(32) comment '分类类型',
	primary key(id)
)auto_increment = 1;

create table reagents_(
	id int(11) not null auto_increment comment '功能剂id',
	cn_name varchar(50) comment '中文名',
	ali_name varchar(50) comment '别名',
	model varchar(255) comment '型号',
	standard varchar(255) comment '规格',
	company varchar(32) comment '生产单位',
	category_id int(10) comment '类型',
	primary key(id)
)auto_increment=1;

create table article_(
	id int(11) not null auto_increment comment '文章id',
	title varchar(50) comment '文章标题',
	content longtext comment '文章内容',
	create_time datetime comment '创建时间',
	admin_id int(11) comment '创建者',
	primary key(id)
)auto_increment=1;
